# アーキテクチャと設計判断の記録

このドキュメントは、テンプレート設計時の主要な技術選定と、その背景・理由を記録するものです。

## 全体構成

クリーンアーキテクチャに基づく4層構造を採用しています。

### 各層の責務

**domain**
ビジネスルールを持つエンティティを配置します。フレームワーク（Spring）やDB（JPA）に関する記述を一切含めません。`@Entity` などのアノテーションも付けません。純粋なJavaのクラスだけで構成し、外部依存ゼロを徹底します。

**usecase/port/in**
アプリケーションが提供する操作をインターフェースとして定義します。Controller からはこのインターフェース経由で UseCase を呼び出します。インターフェース分離の原則に従い、1ユースケース1インターフェースとしています。

**usecase/port/out**
UseCase が外部に求める機能をインターフェースとして定義します。`UserRepository` や `PasswordHasher` がここに含まれます。**依存性逆転の核心部分**で、UseCase 層が具体的な実装ではなくインターフェースに依存する構造を作っています。

**usecase/interactor**
`port/in` で定義したインターフェースの実装です。実際のビジネスフローを記述します。

**adapter/in/web**
HTTP リクエストを受け取る Controller を配置します。リクエスト・レスポンス用の DTO も同梱します。グローバル例外ハンドラーで例外を HTTP ステータスに変換します。

**adapter/out/persistence**
DBアクセスの実装を配置します。`UserJpaEntity`（JPAエンティティ）、`UserJpaRepository`（Spring Data JPA）、`UserRepositoryImpl`（`usecase/port/out/UserRepository` の実装）の3クラス構成です。

**adapter/out/auth**
認証関連の実装を配置します。現状は `BCryptPasswordHasher` のみです。フェーズ2で JWT 関連の実装を追加予定です。

**config**
Spring の設定クラスを配置します。

## 主要な設計判断

### なぜクリーンアーキテクチャか

レイヤードアーキテクチャは依存方向の制御が緩く、UseCase が DB 実装に依存する形になりがちです。クリーンアーキテクチャでは依存性逆転を徹底することで、内側の層（ビジネスロジック）が外側（DB、フレームワーク）に依存しない構造を作ります。

ヘキサゴナルアーキテクチャとの選択も検討しましたが、思想は同じです。ここでは「学習目的」を重視してクリーンアーキテクチャを採用しました。

### なぜドメインモデルとJPAエンティティを分けるか

`domain/model/User` にJPAのアノテーションを付けると、ドメイン層がフレームワークに縛られます。両者を分けて変換コストを支払う代わりに、ドメイン層を綺麗に保ちます。

### なぜ User クラスを不変オブジェクトにするか

複数の処理が同じインスタンスを共有しても予期せぬ変更が起きないようにするためです。更新時は新しいインスタンスを作り直す形になり、コードはやや冗長になりますが、マルチスレッド安全性と副作用の防止という恩恵が大きいです。

### なぜパスワードハッシュ化を独立したインターフェースにしたか

認証認可全体を `AuthProvider` という1つのインターフェースで扱う設計も検討しましたが、パスワードハッシュ化は認証フロー全体と独立した責務です。インターフェース分離の原則に従い、`PasswordHasher` として切り出しました。フェーズ2で `AuthProvider`（JWT関連）を追加する際にも、`PasswordHasher` はそのまま残ります。

### なぜ Repository の戻り値で Optional を使い、UseCase では使わないか

Repository 層では「DBにデータがあるかないか」が日常的な関心事なので `Optional` で表現します。一方、UseCase 層では「指定したIDのユーザーが存在しないこと」を**例外的な状況**として扱い、`UserNotFoundException` を投げる設計にしました。これにより、Controller が「Optional をチェックして404を返す」というロジックを毎回書く必要がなくなります。

### なぜ JWT で RS256 を選んだか（フェーズ2予定）

将来 Supabase Auth 等へ移行する際の互換性を考慮しました。Supabase Auth も RS256 を使用しているため、署名アルゴリズムを揃えておくことで移行コストが下がります。

### なぜ PostgreSQL のコンテナを 5433 で動かすか

ローカル環境に PostgreSQL がインストールされている場合、5432 が競合します。テンプレートとして「他プロジェクトのローカル PostgreSQL に影響を与えない」ことを優先し、5433 で動かしています。

### なぜ Flyway を使い、`ddl-auto: validate` にするか

`ddl-auto: update` は便利な反面、意図しないテーブル変更が発生する危険があります。Flyway でスキーマを明示的に管理し、JPA には「整合性チェックだけ」を任せる構成にしました。本番運用を見据えた構成です。

### なぜ Lombok を使わないか（現状）

学習目的では、まずは素のJavaで書いて構造を理解し、慣れてから `@Getter` などで省略する方が、何が省略されているかを把握できるためです。将来的にリファクタリングで導入する余地はあります。

## 今後の方針

- 認証認可は Spring Security + JWT で自前実装。将来 Supabase Auth 等のクラウドサービスに移行できる構造を維持する。
- フロントエンドは React + TypeScript を予定。モノレポ構成で `frontend/` ディレクトリに配置する。
- テストはユースケース単位のユニットテストと、Controller の結合テスト（MockMvc）を最低限揃える方針。
